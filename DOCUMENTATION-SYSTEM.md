# The Simatrix Documentation System
## A Reusable Methodology for Solo and Small-Team Software Projects

> Built during the Simatrix project (June 2026). Reuse this on any project where
> you want Claude Code and human contributors to produce consistent, documented,
> auditable output — without a senior engineer looking over everyone's shoulder.

---

## What Problem This Solves

Every software project eventually hits the same wall: the person who made a decision
leaves, forgets, or gets asked "why is it done this way?" and can't remember. On a
small team with AI assistance, the problem is worse — every new Claude Code session
starts with zero memory of previous sessions, and two people using AI independently
will produce inconsistent output unless both AIs are reading the same instructions.

This system solves three problems simultaneously:

- **Memory** — decisions made months ago are findable and reasoned
- **Consistency** — two contributors (human or AI) produce identical output
- **Onboarding** — a new contributor can get up to speed without asking anyone

---

## The Core Insight

Documentation is only trustworthy if it was built by reading the actual code,
not by describing what you intended to build. Every file in this system was
generated by having Claude Code read source files first, then write documentation
that matches what the code actually does. Where the docs and code disagreed,
the code won.

**The principle: ground truth lives in the code. Documentation is a map of
that truth, not a plan for it.**

---

## The Six Files

Every project using this system starts with six files at the root level:

```
ProjectRoot/
├── ARCHITECTURE.md    What the system is
├── DECISIONS.md       Why it is that way
├── RULES.md           What you must never change
├── DESIGN.md          What it looks like
├── PRODUCT.md         Who it is for and what it does
└── MODULE-STARTER.md  How to create new modules or topics
```

If the project later grows to cover more than one product family or engineering discipline —
Simatrix started with one (Engineering Graphics) and grew a platform layer for every family after
it — two more root files earn their place:

```
ProjectRoot/
├── PLATFORM-RULES.md          The subject-agnostic subset of RULES.md
└── CLAUDE.module-template.md  A fill-in-the-blanks CLAUDE.md for a brand-new family
```

Plus per-module files that already exist in most projects:
```
ModuleFolder/
├── CLAUDE.md          How the AI behaves in this module
├── CHANGELOG.md       What changed and when
└── DESIGN.md          Module-specific visual decisions (if any)
```

Every module folder also maintains its own CHANGELOG.md (what changed and when). The root-level
CHANGELOG.md tracks changes that span the whole repository. These are not part of the six root
files above but are assumed to exist in every module and at the root.

## What Each File Is For

### ARCHITECTURE.md — The Map
Answers: *What is this system made of and how do the pieces connect?*

Contains:
- What the product is (one paragraph, plain language)
- Folder structure with descriptions
- Component breakdown: every major file, what it does, what it depends on
- Data flow: how a user action reaches the output (diagram with arrows)
- Communication boundaries (APIs, iframes, postMessage contracts)
- What is shared across codebases vs intentionally different
- Known structural issues (honest, not polished)

Written for: a new contributor who has never seen the codebase.
Updated when: folder structure changes, new major files added, relationships change.

---

### DECISIONS.md — The Diary
Answers: *Why is it done this way?*

Contains ADR entries (Architecture Decision Records). Each entry has:
- **Decision:** one sentence, what was chosen
- **Why:** the specific problem it solves
- **Alternatives rejected:** what else was considered and why it lost
- **Consequences:** what this makes easier or harder
- **Status:** Active | Superseded by ADR-XXX | Known issue

The most valuable entries are **reversed decisions** — choices that were made,
tried, and replaced. These prevent future contributors from re-introducing
approaches that already failed.

Written for: anyone who wonders "why on earth is it done this way?"
Updated when: a non-obvious decision is made, or a previous decision is reversed.

---

### RULES.md — The Rulebook
Answers: *What must I never do?*

Every rule is either ✅ DO or ❌ NEVER. No "consider" or "try to" language.
Every rule cites its source ADR. Rules are grouped by category:
- Master codebase and deployment rules
- Platform and runtime rules
- Architecture rules
- UI and visual rules
- Camera and animation rules (if applicable)
- Answer validation rules (if applicable)
- Cross-module harmony rules
- Documentation rules
- Anti-patterns (the quick-reference ❌ NEVER list)

This file fulfills the role of a standards document — the thing you reference
before starting any task to check what constraints apply.

Written for: Claude Code and human contributors before every task.
Updated when: a new rule is established, or a mistake recurs and needs enforcement.

---

### DESIGN.md — The Visual Contract
Answers: *What does it look like and why?*

Contains:
- Creative North Star (the design metaphor that drives all decisions)
- Color system: every token, its value, its semantic meaning
- Typography: fonts, sizes, weights, when each is used
- Spacing and layout principles
- Component standards: panels, buttons, inputs, steppers
- Cross-module consistency rules
- Module-specific exceptions
- Known gaps and open questions

Critical: every token value is **verified against the actual code**, not copied
from a design tool. Where the design doc and the code disagree, the code wins
and the doc is updated to match.

Written for: anyone implementing UI, and AI assistants that need to match the
existing visual language.
Updated when: new tokens are added, components are standardized, or the audit
finds drift between modules.

---

### PRODUCT.md — The Strategic Contract
Answers: *Who is this for and what must it do?*

Contains:
- What the product is (platform-level)
- Who it is for (primary persona, usage contexts)
- What it is not (anti-references — deliberate boundaries)
- Core design principles with enforcement cross-references
- Feature list per module with status: EXISTS | PLANNED | PARTIAL
- Accessibility commitments with implementation status
- Planned but not yet implemented
- Open questions

Critical: every feature claim is **verified against the code**. Features that
exist are marked EXISTS. Features that are planned but not built are marked
PLANNED. Nothing is implied to exist if it doesn't.

Written for: product decision-making, new contributors understanding scope,
AI assistants understanding what the product should and should not do.
Updated when: new features are built or planned, scope changes.

---

### MODULE-STARTER.md — The Onboarding Playbook
Answers: *How do I create a new module or topic without breaking anything?*

Contains:
- Mandatory reading list before starting
- Decision tree: which case applies to this new module?
- Step-by-step checklist for each case (copy / adapt / create fresh)
- File-by-file table: which files are shared contracts vs topic-specific
- Verification checklist before shipping
- Common mistakes section

Critical: the "copy vs adapt vs create fresh" distinction is **verified by
md5sum**, not assumed. Files that are byte-identical across modules are in the
"copy exactly" category. Files that diverge are in the "adapt" category.

Written for: anyone starting a new module, topic, or subject area.
Updated when: new module patterns are established, or a new mistake is discovered.

---

### PLATFORM-RULES.md — The Subject-Agnostic Foundation
Answers: *What must every module do, no matter which engineering discipline it teaches?*

Contains:
- The subject-agnostic subset of RULES.md — every rule that named no specific codebase file or
  lesson, extracted and generalized where the original wording used a subject-specific example
- The same ✅ DO / ❌ NEVER format as RULES.md, each rule still citing its source ADR or shared doc

How it relates to RULES.md: RULES.md is specific to your original product family (in Simatrix,
Engineering Graphics). PLATFORM-RULES.md is what every other family inherits as its foundation —
a new family's own RULES.md is written on top of it, never duplicated or contradicted.

Written for: new subject modules that do not inherit the original family's RULES.md.
Updated when: a rule is found to be platform-wide rather than family-specific, or a new platform
contract is established (e.g. a data-format rule discovered from a live bug).

---

### CLAUDE.module-template.md — The New-Family Starter
Answers: *What does a brand-new subject module's CLAUDE.md start from?*

Contains:
- A fill-in-the-blanks CLAUDE.md carrying the platform-wide contract sections pre-filled — the
  root-doc pointer block, the session digest protocol, the "keeping documents current" block —
  with TODO placeholders for the family-specific architecture rules only that family can decide

How it relates to CLAUDE.md: the per-module CLAUDE.md for a new family is created by copying this
template and filling in the blanks. It deliberately does **not** point at the original family's
own ARCHITECTURE.md, DECISIONS.md, or RULES.md — those describe decisions specific to a discipline
the new family doesn't share.

Written for: anyone starting a new subject module.
Updated when: the platform-wide sections of CLAUDE.md change (new session digest format, new root
doc pointer, new platform rule added).

## The Three-Step Creation Process

These six files cannot be written from memory or design intent. They must be
built by reading the actual codebase. Use this three-step process:

### Step 1 — ARCHITECTURE.md first

Prompt Claude Code to:
1. Read all session memory files at `~/.claude/projects/`
2. Read all existing documentation files (CLAUDE.md, DESIGN.md, etc.)
3. Read every source file in every module
4. Write ARCHITECTURE.md with all nine sections
5. Report: which files were read, which sections felt thin, any contradictions

**Do not skip the memory file reading.** Those files contain development history
that is not visible in the source code — reversed decisions, abandoned approaches,
hard-won lessons.

### Step 2 — DECISIONS.md second

Prompt Claude Code to:
1. Read ARCHITECTURE.md (already written — do not re-read all source files)
2. Read session memory files again for decision evidence
3. Read both CHANGELOG.md files for "why" entries
4. Read CLAUDE.md files for embedded rationale
5. Write 10–20 ADR entries, prioritizing reversed decisions
6. Report: which categories had most evidence, which were thin, reversed decisions found

**The pause-and-confirm rule:** Claude Code should report its findings before
writing any ADR, so you can correct misunderstandings before they get committed.

### Step 3 — RULES.md third

Prompt Claude Code to:
1. Read ARCHITECTURE.md and DECISIONS.md (both already written)
2. Translate every ADR consequence into a ✅ DO or ❌ NEVER rule
3. Translate every §9 structural issue into a rule
4. Resolve any dangling references in existing CLAUDE.md files
5. Update all CLAUDE.md files to reference RULES.md

Then separately: DESIGN.md and PRODUCT.md each follow a two-phase prompt —
audit first, consolidate second, with a mandatory human confirmation pause
between the two phases.

### When a project spans more than one product family

This three-step process (ARCHITECTURE → DECISIONS → RULES) applies both to your original product
family and to any new family you add later. The difference for a new family:

- It starts from PLATFORM-RULES.md, not RULES.md.
- It uses CLAUDE.module-template.md as its CLAUDE.md starting point, not a copy of an existing
  module's CLAUDE.md.
- Its RULES.md extends PLATFORM-RULES.md rather than being built from scratch — write only the
  rules PLATFORM-RULES.md doesn't already cover, on top of it as the stated foundation.

The resulting ARCHITECTURE.md, DECISIONS.md, and RULES.md are new, local files scoped to the new
family — not entries added to the original family's root files.

## The Audit-Then-Confirm Workflow

For DESIGN.md and PRODUCT.md specifically, always use this two-phase approach:

**Phase 1 — Audit (read only, no file changes):**
Claude Code reads the existing documentation and the code.
It produces an audit report: accurate claims, outdated claims, missing claims,
contradictions. It then **stops and waits for your confirmation.**

**Phase 2 — Consolidate (file changes only after confirmation):**
You review the audit report. You correct any misunderstandings.
Then Claude Code writes the consolidated file based on the confirmed audit.

**Why this matters:** Claude Code might flag something as "outdated" that you
intentionally kept for a reason, or "missing" when it's actually planned and
documented elsewhere. The confirmation pause lets you catch these before they
become permanent mistakes in the documentation.

## How CLAUDE.md Connects Everything

Every module folder has a CLAUDE.md that tells Claude Code how to behave. For this system to
work, every CLAUDE.md must point back at the shared root docs — but *which* root docs it points
at depends on whether the module belongs to your original product family or is the first module
of a new one.

**Case A/B — a module within your original product family** (in Simatrix: an Engineering
Graphics topic). Its CLAUDE.md carries this block, pointing at all five original root docs:

```markdown
## Project-wide documentation (read before cross-module tasks)
Before starting any task that touches shared behavior, UI patterns,
or cross-module consistency, read these root-level files:
- ../ARCHITECTURE.md  — system map, component breakdown, data flow
- ../DECISIONS.md     — why key decisions were made (ADR log)
- ../RULES.md         — what you must and must not do (enforcement)
- ../DESIGN.md        — color tokens, typography, component standards
- ../PRODUCT.md       — who it's for, features, accessibility commitments
```

**Case C — the first module of a new product family** (in Simatrix: a new subject module such as
Mechanical, Electrical, or Civil). Its CLAUDE.md, copied from CLAUDE.module-template.md, points at
only three root docs:

```markdown
## Project-wide documentation (read before cross-module tasks)
Before starting any task that touches shared behavior, UI patterns,
or cross-subject consistency, read these root-level files:
- ../DESIGN.md         — color tokens, typography, component standards (the platform design system)
- ../PRODUCT.md        — who it's for, features, accessibility commitments (the platform product contract)
- ../PLATFORM-RULES.md — what every module must and must not do, regardless of subject
```

— plus the new family's own local ARCHITECTURE.md, DECISIONS.md, and RULES.md, which it writes
itself rather than inheriting. A new-family module must **not** point its CLAUDE.md at the
original family's ARCHITECTURE.md, DECISIONS.md, or RULES.md — those describe decisions and rules
specific to a discipline it doesn't share.

Either block is what makes two separate Claude Code sessions — your teammate's and yours —
behave identically. Without it, each session reads only the module-level docs and makes
independent decisions that will diverge.

Also add this block for automatic document maintenance:

```markdown
## Keeping root documents current
After completing any task, check whether the work involved:
- A non-obvious decision between two real options
  → add an ADR to ../DECISIONS.md
- A reversed or superseded previous decision
  → update the relevant ADR status and add a new one
- A new rule that must be enforced going forward
  → add it to ../RULES.md with its source ADR cited
- A structural change to the codebase
  → update ../ARCHITECTURE.md Section 2 or 3

Only update when the change has architectural significance.
Routine bug fixes do not need ADRs.
```

## The Source vs Note Decision Rule for NotebookLM

If you use NotebookLM as a second brain alongside Claude Code:

**Keep as permanent Source:**
- All root documentation files — the original six, plus PLATFORM-RULES.md and
  CLAUDE.module-template.md once your project has grown to support more than one product family
- DESIGN.shared.md or equivalent (the cross-module visual contract)
- Evergreen best-practices docs you uploaded

**Keep as rotating Note (one per sprint, delete after two sprints):**
- Session digests
- Task breakdowns for a specific sprint
- Claude responses containing a decision worth preserving

**Never store:**
- Raw implementation code (lives in the repo)
- Completed task lists with no architectural content
- Intermediate prompt drafts

**The practical test:** For any old source, ask "is this information now in one
of the six root files?" If yes, delete it. A NotebookLM with six clean sources
answers better than one with twenty sources where half are stale.

## The Session Digest Protocol

Add this to CLAUDE.md to get structured end-of-session output that is short
enough to paste into NotebookLM without uploading a file:

```markdown
## Session digest
At the end of every session, produce a digest in this format:

### SESSION DIGEST — [date] — [feature/task]
**What changed:** (3–5 bullets, concrete)
**Decisions made:** (with brief rationale)
**Patterns introduced:** (reusable code patterns or conventions)
**Open questions / next steps:**
**Files modified:** (list)
```

## The Verification Checklist (Before Sharing With a Teammate)

Run this before declaring the documentation system ready to share:

1. **No dangling references** — every file referenced in any doc actually exists
2. **No contradictions** — RULES.md and DECISIONS.md agree on every point
3. **CLAUDE.md completeness** — every module CLAUDE.md has the correct root doc pointer block —
   Case A/B modules point at all five root docs (ARCHITECTURE.md, DECISIONS.md, RULES.md,
   DESIGN.md, PRODUCT.md); Case C modules point at only three (DESIGN.md, PRODUCT.md,
   PLATFORM-RULES.md) plus their own local three-doc set
4. **Open items tracked** — every ⚠️ PLANNED and "needs review" flag has a clear
   owner and next action
5. **MODULE-STARTER.md gap check** — a teammate could follow it step-by-step
   and produce a working new module without asking you anything

Run this verification prompt in Claude Code:

```
Read all root documentation files and both module CLAUDE.md files.
Check for:
1. Dangling references — files referenced that do not exist
2. Contradictions — rules that conflict with ADRs
3. CLAUDE.md completeness — root doc pointer blocks present
4. Open items — list every flag with owner and next action
5. MODULE-STARTER.md gap check — identify any step too vague to follow

Do not modify any files. Report findings only.
```

## What Makes This System Different From Just Writing Docs

Most documentation is written from memory or design intent and goes stale
within weeks. This system has three properties that prevent that:

**1. Code-grounded from the start.**
Every claim was verified against the actual source files. Token values were
checked in the CSS. Features were confirmed in the JS. File relationships were
confirmed with md5sum. The documentation cannot contradict the code because
the code was read first.

**2. Self-referential enforcement.**
RULES.md cites DECISIONS.md. DECISIONS.md cites ARCHITECTURE.md. CLAUDE.md
points at all of them. When Claude Code works in any module, it reads the
whole chain. A rule written once is enforced everywhere automatically.

**3. Honest about gaps.**
Every file has a section for open questions and known issues. "Could not
confirm from code — needs review" is a valid and encouraged phrase. A
documentation system that pretends to be complete is more dangerous than one
that clearly marks its own boundaries.

## Sharing the System With New Contributors

### What to hand a teammate starting a new subject module

If a teammate is starting the *first* module of a new product family — not another topic in your
original family — hand them only the platform layer, not your family's own internals:

- `DESIGN.md`
- `PRODUCT.md`
- `PLATFORM-RULES.md`
- `CLAUDE.module-template.md`
- `DOCUMENTATION-SYSTEM.md` itself (so they understand the methodology, not just its output)

**What not to share** (specific to your original family — in Simatrix, Engineering Graphics):
- `ARCHITECTURE.md`
- `DECISIONS.md`
- `RULES.md`
- `MODULE-STARTER.md` (its Case C section is genuinely useful, but the file's other sections are
  Engineering-Graphics-specific copy/adapt mechanics that will mislead a contributor building
  something else — point them at `CLAUDE.module-template.md` + `PLATFORM-RULES.md` instead, which
  cover the same ground without the family-specific packaging)

Handing over the wrong set is exactly how a new family ends up quietly depending on decisions —
a specific copy-deploy model, a specific rendering engine, a specific answer-validation scheme —
it never agreed to and doesn't need.

### Two harmonisation scenarios

**1. Starting fresh — the day-one starter prompt pattern.**
Before a new family writes any code, put its CLAUDE.md's "subject-specific architecture rules"
placeholder in front of Claude Code as a starter prompt — it is five architecture questions, not
a form to skip:

1. What renders the module — and if it's 3D, which structural pattern (a single orchestrator +
   leaf modules, a shared engine + thin pages, or something new) — and why?
2. What is the single state-change pipeline from "the learner changed something" to "the scene
   reflects it," and what is its disposal/cleanup contract?
3. What are the module boundaries — which files own what, and what may or may not import what
   else?
4. What domain-specific viewport encodings does this subject need on top of the shared design
   tokens?
5. If the module teaches via exercises, where does the answer/validation logic live and how
   tolerant is it?

Answering these *before* writing simulation code, and recording the answers as the new family's
first ADRs, is what keeps it from drifting into an undocumented architecture as it grows.

**2. Already has code — the audit-then-harmonise pattern.**
If a teammate already built something before adopting this system, don't retrofit the root files
from memory. Run the same two-phase discipline as the Audit-Then-Confirm Workflow above, applied
to reconciling existing code against the platform contract:

- **Phase 1 — Audit (read only):** Claude Code reads the existing code against PLATFORM-RULES.md,
  DESIGN.md, and PRODUCT.md and produces a findings table — what complies, what doesn't, what's
  ambiguous. It stops and waits for confirmation before changing anything.
- **Phase 2 — Fix, in severity order:** once confirmed, fixes are made in three passes, not all
  at once:
  1. **CRITICAL** — platform-contract violations (a build step, an unpinned dependency,
     `postMessage` usage, a second reset path, a missing `meta.json` field)
  2. **IMPORTANT** — design/rule violations that don't break the platform contract but create
     drift (a hard-coded hex, a wrong font weight, a missing two-cue pairing)
  3. **DOCUMENTATION** — write the family's own ARCHITECTURE.md / DECISIONS.md / RULES.md to
     describe what the audit found, and add any missing ADRs for decisions the code already made
     implicitly

This ordering matters: fixing documentation before the CRITICAL code issues just documents a
contract violation instead of removing it.

### The platform-wide bug fix pattern

Some bugs aren't local to one module — they're a platform contract violated the same way
everywhere. (Simatrix's concrete example: the `meta.json` `difficulty` field was found shipping
with capitalised values like `"Intermediate"` across every module, when the backend requires
exactly `beginner` / `intermediate` / `advanced`, all lowercase.) Fixing the live instances
without fixing the *rule* just means the same bug ships again in the next module. The pattern:

1. **Find all affected files first** — search the whole repo; do not assume which folders are
   affected. (The meta.json fix searched every module's `meta.json`, not just the one where the
   bug was noticed, and found six.)
2. **Audit before fixing** — produce the findings table and wait for confirmation, same discipline
   as the Audit-Then-Confirm Workflow above.
3. **Fix the code** — correct every affected file found in step 1.
4. **Add a rule to PLATFORM-RULES.md** to prevent recurrence, citing the bug as its source
   (Simatrix's example: PLATFORM-RULES.md §1.11a, cited as "Live bug found 2026-07-02").
5. **Mirror the same rule to RULES.md** — or to a family-specific RULES.md if the bug only
   affects one family — so the platform layer and the family layer state the same rule (Simatrix's
   example: RULES.md §2.11a mirrors PLATFORM-RULES.md §1.11a verbatim).
6. **Add an anti-pattern entry to PLATFORM-RULES.md Section 5** (the 60-second scan) so the rule
   surfaces in the quick pre-flight check, not only in the numbered rule list.
7. **Record it in the root CHANGELOG.md** — what broke, what was fixed, and which rule now
   prevents it.

## Reuse Checklist for a New Project

When starting a new project, run the three-step creation process in order.
These are the things to adapt for your new context:

- [ ] Identify your "master codebase" equivalent (the Module2 role)
- [ ] Identify your deployment pattern (the topic folder copying pattern)
- [ ] Identify your shared contracts (the DESIGN.shared.md equivalents)
- [ ] Identify your communication boundary (the window.simAPI equivalent)
- [ ] Write the creative North Star for DESIGN.md (the "Patient Tutor's Paper"
      equivalent — what metaphor drives your visual decisions?)
- [ ] Define your primary persona for PRODUCT.md (the "struggling first-year"
      equivalent — who are you optimizing every decision for?)
- [ ] Set up CLAUDE.md in every module with the root doc pointer block
- [ ] Add the session digest protocol to CLAUDE.md
- [ ] Create the six root files using the three-step process
- [ ] Copy CLAUDE.module-template.md as the new module's CLAUDE.md and fill in all placeholders
      before writing code
- [ ] Write the new module's RULES.md as an extension of PLATFORM-RULES.md — start from the
      platform foundation, add subject-specific rules on top
- [ ] Add the meta.json difficulty field using exactly lowercase values: beginner | intermediate
      | advanced
- [ ] Run the verification checklist before sharing with teammates
